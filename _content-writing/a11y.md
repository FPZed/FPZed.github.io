# Why Accessiblity?
This is a guide for content writers on our team to write more accessible content for our products. We'll go over best practices and user needs, and learn how developers, designers, and end-users interact with accessible content and technology.

## What is web accessibility?
Web accessibility is the practice of designing user experiences that accommodate people with disabilities, such as visual, auditory, cognitive or motor disabilities. This is accomplished through thoughtful design and assistive technology.

The goal of an accessible website is for disabled users to gain the same value from the website as abled users do.

## Why make things accessible?
### Good user experience
Good design and web accessibility go hand in hand. Websites that are accessible work better for all users.

<p style="margin-bottom: 0px">Accessibility also helps people without disabilities:</p>
* Older people with changing abilities due to aging.
* People with slow internet connections or limited bandwidth.
* People with temporary or situational disabilities. For example, someone with a broken arm, or someone in a library where they can’t play audio.

<blockquote>“Accessibility, usability, and inclusion are closely related aspects in creating a web that works for everyone.”

### Legal mandate
Many countries have passed laws or regulations that require public and private organizations to make their public-facing web content accessible. In the United States, it is illegal to discriminate against people with disabilities, and neglecting accessibility could lead to a lawsuit.

In Ontario, it will be legally required that private companies follow international web accessibility standards by 2021.

# User Needs
You need to anticipate different users' needs and methods of interaction. When planning out content, keep in mind what types of disabilities people have and how that affects their experience.

## Visual disabilities
### Blindness
#### Challenges
* Can't rely on images, photos or graphic illustrations for information.
* Can't navigate using a mouse.

#### Tools and Techniques
* Uses a Screen Reader to listen to text.
* Uses a Refreshable braille display to read.

#### Solutions
* Develop a website that is friendly to screen readers. See Semantic Structure.
* Provide alternative text for images, graphics, icons and buttons.
* Create audio descriptions for video.
* Make sure your writing sounds natural when read aloud.

### Colour blindness
#### Challenges
* Can't distinguish between some colours.

#### Solutions
* Use sufficiant contrast between colours. See - link to contrast checkers.
* Do not rely on colour alone to convey information.

### Low vision
#### Challenges
* Can't make out small details.

#### Tools and techniques
* Uses a screen magnifier to focus on specific parts of the page.
* Adjusts the zoom on their browser.

#### Solutions
* Avoid horizontal scrolling to accomodate zoomed in browsing.
* Limit using text within a raster image file like PNG or JPG, as the text may not look clear enlarged.
* Line length should be between 50 and 75 characters per line.

### Dyslexia
#### Challenges
* Has difficulty reading and writing.
* Mixes up words.
* Reads slowly.

#### Tools and techniques
* Uses a Screen Reader to listen to text.
* Types using Voice Recognition to record speech-to-text.

#### Solutions
* Avoid writing in all caps or using italics. It is more difficult to make out individual letters.
* Avoid timed interactions, or allow the user to give themselves more time.

## Auditory disabilities
### Deafness
#### Challenges
* Can't use audio to get information.

#### Tools and techniques
* Uses closed captioning to watch videos.

#### Solutions
* Include transcripts alongside audio recordings or video.
* Add synchronous captioning to videos.

## Other disabilities
### Cognitive
#### Challenges
* Learning disability or intellectual disability makes it difficult to process information.
* Confused by complex layouts.
* Difficulty reading long chunks of text and long, wordy sentences.

#### Solutions
* Write content for a sixth to eighth grade reading level.
* Supplement text with illustrations or video.
* Avoid timed interactions, or allow the user to give themselves more time.
* Use logical, descriptive headings to group together information.
* Organize information into small, manageable chunks.
* Interactions should be consistent and predictable.

### Neurological
#### Challenges
* Prone to seizures

#### Solutions
* Avoid flashing animations, or give the option to turn them off.

### Motor
#### Challenges
* Can't navigate using a mouse.

#### Tools and techniques
* Uses a keyboard to navigate.
* Types using Voice Recognition to record speech-to-text.
* Uses Sip-and-Puff technology to navigate. See Sip-and-Puff switch.

#### Solutions
* Develop products so they can be navigated by keyboard.
* Let users skip over long lists and navigation. See Skip links.

# Assistive Technology
Assistive technology lets users to interface with the web. The web, in turn, must be designed to interface with assistive technology.

## Screen reader
A **screen reader** is software that uses text-to-speech technology to scan computer applications, such as an internet browser, and reads out content as the user navigates.

Using their keyboard, the user focuses on elements of a web page, such as the page title, a menu, buttons, text or images. These are described aloud by the screen reader.

### How screen readers work
For a web page to successfully interact with screen readers, it needs to be designed so that users can navigate in a logical, linear way. A web page must be developed so that screen readers can accurately describe the elements inside the web page.

### Semantic structure
Developers build websites with HTML. They structure websites with elements such as `header`, `nav` (the navigation menu), `section` and `footer`, along with `img` (an image), `table`, and `p` (paragraphs). These are called semantic because the tag itself describes what the element is. The opposite is a non-semantic tag such as `<div>`.

When a screen reader scans a website, it reads this HTML structure so that it knows how to navigate and describe it. If a website is well structured and semantic, it can be easily read by both humans and computers.

(note) Strong and EM HTML tags semantically add emphasis to text. However, there are some drawbacks. Many assistive technologies do not call these tags out by default. Many users report nuisance because these tags are commonly overused.

### ARIA
Assistive Rich Internet Applications (ARIA) is a set of HTML attributes that improve interactivity by screen readers. By labelling HTML elements with roles such as "button" or "banner," it gives users a clear idea of what the screen reader is interacting with. ARIA has no effect on the web page outside of assistive technology interactions.


### Alternative text
Alternative text is used when a screenreader needs to describe a non-text element such as an image or button. For a user to get all the information they need, the alt-text should convey either the content of the element (what it means) or its function (what it does).

Developers accomplish this by using the `alt` attribute for HTML elements or ARIA labels.

See: “alternative text” on page 24

### How users navigate with screen readers
Screen reader users navigate by keyboard in a **linear progression**: up to down and left to right. They move step-by-step, forwards or backwards, and can also skip across the website by jumping between headers, links or other elements. "Skip links" are invisible links only detectable by screen readers, and allow users to quickly jump to the main content, bypassing titles and navigation menus.

This unique method of navigation should affect how you write. Following these rules will also make your writing snappier and more meaningful.

| Navigation method                                      | What it means for you                                |
|--------------------------------------------------------|------------------------------------------------------|
| Readers progress through a page one step at a time.    | Unlike other users, screen reader users can't glance over the page to see what's ahead. |
| Readers progress up to down and left to right.         | Important information should be presented to the left or above. A form's label should be placed above it, so the user reads it first. |
| Readers jump from heading to heading to quickly navigate and skim the content to find the part they're interested in. | Headings should stand alone and accurately represent the content underneath them, without help from other nearby text. |
| Readers only listen to the first words of a heading or paragraph before skipping ahead. | Headings, labels and lists shouldn't repeat themselves with "How to's" or preamble, so that meaning can be gleaned as early as possible. |
| Readers skip over paragraphs, listening to the first or second sentence before moving on. | Make sure the first sentence of each paragraph contains its key point. |
| Readers jump from link to link.                        | Users might ignore non-link content, so the link itself should accurately describe what the link will do. |

### How screen readers read different characters
The sound and speed of the screen reader voice can be customized by the user. The language the voice speaks, or the way it pronounces letters, is determined in HTML using the `Lang` attribute.

A screen reader will announce the **page title** when first loading a web page. It also tells the user where they are on the page to give them a bearing.

#### Pauses
Screen readers will briefly pause for:
* Periods **.**
* Semi-colons **;**
* Commas **,**
* Question marks **?**
* Exclamation points **!**
* Paragraph endings **¶**

#### Acronyms
Screen readers will atempt to read out acronyms if the word is sufficiently pronounceable. Otherwise, the letters are spelled out.
* NASA is prounounced "NAH-suh."
* FBI is prounounced "F. B. I."
<p>The <abbr title = "Hypertext Markup Language">HTML</abbr> tag `abbr` provides the long form of an abbreviation or acronym which some screen readers will read out.</p>

#### Symbols & punctuation
Which symbols or punctuation are read out differ betwen screen readers and can be customized by the user.
Symbols such as ©, @, $ and % are generally always read out.
Punctuation symbols such as quotes, question marks and dashes are often unspoken, and can make the screen reader pause or make a voice inflection.

#### Images and links
Screen readers will read alternative text for images and links. Images, links and buttons will be called out as such by the reader, so you shouldn't include "image of" or "link to" in the alt-text. (This needed here?)

## Screen magnifier
Screen magnifiers are a popular assistive technology that enlarge content, especially text, so that people with visual impairments can see better. Modern accessibility standards require websites to be compatible with screen magnification software. Text must be zoomable up to 200% without loss of website functionality.

Accommmodation for screen magnifier users mainly comes down to website design. However, there are few things a writer should keep in mind:
* Avoid columns: Users with screen magnifiers are more likely to miss extra columns. In general, position relevant and related content in a vertical form.
* Avoid the need for horizontal scrolling.
* Limit using text within a raster image file like PNG or JPG. These image formats might not scale properly, causing the text to look blurry or pixelated when zoomed in.

## Voice recognition
Voice or Speech recognition is used by users with motor or cognitive disabilities to input text or give commands without needing a mouse or keyboard. It works by listening to their voice with a microphone. This technology can also be used by people with temporary limitations, like tendinitis, carrying a small child or animal while working, or by people who like to multi-task!

Labels and alt-text for buttons or any other type of control need to accurately describe their function so that it's clear which voice command the user should speak. The visible label must also match the accessible name in the website's code. For example, the accessible name could be the HTML text, or an ARIA label. If the visible label says "Go," but the ARIA label says "Submit," then the Voice recognition will not work.

## Refreshable braille display
A refreshable braille display is a mechanical-electronic hardware that dynamically outputs braille characters. It is used together with a screen reader to output a site's text. This is accomplished by raising round-tipped pins through holes on a flat surface to create 8-dot braille cells. The cells form a line of braille that changes as the user parses through text with a keyboard.

This is another good reason to write in plain language and avoid sentences that run too long.

To accomodate braille displays, writers should adhere to the same guidelines as for screen readers.

## Sip-and-puff switch
A sip-and-puff switch lets users interface with a computer by inhaling (called "Sipping) and exhaling (called "Puffing") into a straw or tube. The switch detects air pressure, allowing the user to input a "hard" sip or puff, or a "soft" sip or puff. Using these controls, the user can emulate keyboard, mouse or joystick usage.

Sip-and-puff technology is used by users with motor disabilities who aren't able to use a keyboard or mouse. Using sip-and-puff can be tiring, so it's important to beable to navigate efficiently.
* Websites should be completely navigable by keyboard.
* Provide skip links for the user to skip to useful content.
* Write accurate labels and headers so that the user has a clear idea of where they want to go.

# Standards and laws
Worldwide organizations have set out standards and regulations for web accessibility. Even if it's not mandatory, these are solid guidelines for your own practice. They're also important for legal liability. Note: This section is not meant to give out legal advice.

## WCAG 2.0
The Web Content Accessibility Guidelines (WCAG) are standards set out by the World Wide Web Consortium (W3C) for use by web content developers to create an accessible internet.
WCAG 2.0 (and 2.1) are adhered to by governments and organizations around the world. There are three levels of conformance to the WCAG 2.0 accessiblity requirements:

Level A
A minimal level of conformance for basic usability by disabled users. Alternative text and captions are provided, design is logical and keyboard firendly, and information is accessible in alternative ways.

Level AA
Suggested level of conformance to provide adequate usability by disabled users. Full compatibility with assistive technology, optimal keyboard navigation, headings and labels accurately describe all elements, and measures are taken to help the user avoid errors.

Level AAA
Full conformance that provides an ideal experience for disabled users. These requirements are not always feasible or sensible for a website.
